## Совместный конструктор поездки в реальном времени
### 1) Пользователи и доступ
- Система должна поддерживать регистрацию/вход пользователя (логин/пароль или токен-сессии).
- Пользователь должен иметь возможность создавать поездку.
- Система должна поддерживать роли в рамках поездки:
  - Owner (создатель): управляет доступом, удаляет поездку.
  - Editor: редактирует данные поездки.
  - Viewer (наблюдатель): только просмотр, получает обновления в реальном времени.
- Owner должен иметь возможность пригласить пользователя в поездку (по коду/ссылке/инвайту) и назначить роль.
- Owner должен иметь возможность удалить участника / изменить его роль.
### 2) Реальное время и совместная работа
- При изменении данных поездки одним участником система должна моментально рассылать изменения всем подключённым участникам и наблюдателям этой поездки.
- Система должна поддерживать одновременную работу многих пользователей в одной поездке и во множестве поездок.
- Система должна корректно обрабатывать конкурентные изменения одних и тех же сущностей (например, правки одного пункта маршрута):
  - применение по версии/ревизии,
  - либо разрешение конфликта (отклонить/принять последнее/merge по правилам).
### 3) Данные поездки (что можно хранить и редактировать)
- 3.1 Структура поездки
	- В поездке должны быть базовые поля: название, период (даты/диапазон), описание.
    - Поездка должна содержать дни/этапы (Day/Stage), которые можно добавлять/удалять/переименовывать/переставлять.
- 3.2 Маршрут и точки
  - Каждый день/этап должен содержать список пунктов плана (Place/Activity).
  - Для пункта плана должны поддерживаться поля: название, время (опционально), описание/заметки, тег/категория, ссылка.
  - Должны поддерживаться операции: добавить/изменить/удалить пункт, изменение порядка.
- 3.3 Чек-листы и задачи поездки
	- В поездке должен быть чек-лист задач (например: билеты, страховка, вещи).
  - Для задачи: текст, статус (done/undone), исполнитель (опционально), дедлайн (опционально).
  - Должны поддерживаться операции: добавить/отметить/изменить/удалить.
- 3.4 Бюджет и расходы
	- В поездке должен быть раздел бюджета: валюта, общий лимит (опционально).
  - Пользователи должны уметь добавлять расходы: сумма, категория, кто оплатил, комментарий, дата/день.
  - Система должна считать:
    - итого расходов,
    - по категориям,
    - баланс по участникам (кто кому должен) по простому правилу “поровну” или “доли” (один выбранный режим достаточно).
### 4) История и “запись поездки”
- Система должна сохранять историю изменений поездки (события: кто/когда/что поменял).
- Должна быть возможность открыть журнал событий и увидеть список изменений.
- При подключении клиента к поездке система должна отдавать:
    - текущее состояние поездки или состояние + события с последней версии (для догрузки).
### 5) Коммуникация внутри поездки
- В поездке должен быть комментарий/чат поездки (минимум: сообщения текстом).
- Должны поддерживаться операции: отправить сообщение, получить историю сообщений.
### 6) Поиск, фильтрация, экспорт
- Должен быть поиск по поездке (по пунктам плана/задачам/расходам).
- Должен быть экспорт поездки в файл (например JSON) и/или “краткий план” (текстовый/Markdown) — достаточно одного формата.
- Должен быть импорт из экспортированного формата (минимально: восстановить поездку из JSON).
### 7) Подключения и сессии
- Клиент должен уметь подключаться к серверу, выбирать поездку, получать обновления.
- Система должна поддерживать переподключение клиента без потери целостности состояния (догрузка изменений после разрыва).
